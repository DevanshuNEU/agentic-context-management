# Chapter 25 — Five Approaches to Context Management

**Chapter Type: C** — Framework and Approach Evaluation (Type C)

**The book's master claim** is that architecture is the leverage point, not the model. This chapter is a specific instance of that claim: the choice of who controls context curation — and when, and by what mechanism — determines the failure mode of a long-running agent. A larger context window does not fix this. A better model does not fix this. The architecture does.

**Core claim:** After reading this chapter, a student will understand how the locus of control for context curation determines the failure mode of a long-running agent well enough to select the right context management architecture for a given workload without making the mistake of treating context overflow as a capacity problem solvable by a larger context window.

This chapter evaluates five approaches — Passive Truncation, Compaction-on-Threshold, Recursive Language Models (RLM), Lossless Context Management (LCM), and Agentic Context Management (ACM) — applied to the same workload. Each approach makes a different architectural bet. Each bet produces a characteristic failure mode. Understanding which failure mode your system can live with is the design decision this chapter teaches.

**Learning outcomes** (Bloom's Taxonomy):

1. **Remember:** Name the five context management approaches and the failure mode each produces.
2. **Understand:** Explain why the KV cache's append-only architecture makes context rot structurally inevitable under passive strategies.
3. **Apply:** Given a workload description (session length, dead-end frequency, reversibility requirement), select the appropriate context management approach from the Decision Matrix.
4. **Analyze:** Given a session log, identify the turn at which vocabulary contamination begins to bias hypothesis generation and trace the causal chain to the wrong output.
5. **Evaluate:** Compare the failure signatures of engine-driven and model-driven context management approaches and justify which failure mode a given production system can tolerate.

## The Scenario

At 9:47 on a Tuesday morning, a senior engineer at a mid-sized fintech company opened a terminal and invoked her team's coding agent — call it Archer — and gave it a single instruction: *Find and fix the authentication bug that's causing intermittent 401 errors on the token-refresh endpoint.* The bug had been evading human review for three days. Logs showed it occurring in roughly one session in forty, always under load, never in staging. She set Archer loose and turned to other work.

Archer began methodically. It read the auth middleware. It traced the token-refresh handler through four layers of abstraction — a JWT decoder, a Redis session cache, a database fallback, and an async refresh lock. It identified two plausible suspects: a race condition in the lock acquisition path, and a clock-skew bug in the token-expiry comparison. It generated hypotheses, wrote test scaffolding to isolate each one, and logged its reasoning as it went. By 11:15, its context window held something that looked, structurally, like good engineering thinking: a record of premises, tests, and partial conclusions.

Then Archer hit its first dead-end. The race condition hypothesis collapsed — the lock was implemented with a Redis `SET NX` call with a proper TTL, and the timing windows were too narrow to explain a one-in-forty failure rate. Archer noted this in its working log and moved on to the clock-skew theory. But it did not discard the dead-end. The failed race condition hypothesis stayed in context — all of its scaffolding code, all of its intermediate reasoning, all of its test output — sitting in the upper portion of the window like furniture left in a room after the party is over. The tokens were still there. They were simply no longer true of the problem being solved.

By 1:30 in the afternoon, Archer had accumulated four hours of session history. The context window was not full — not even close by the metrics that engineers typically watch. But the window's content had quietly rotted. Dead-end A's scaffolding was influencing the probability weights that shaped dead-end B's diagnosis. When Archer began investigating the clock-skew path, it repeatedly generated suggestions that unconsciously echoed the lock-acquisition vocabulary from the abandoned hypothesis: it proposed checking `lock_acquired_at` timestamps rather than `token_issued_at` timestamps, conflating two entirely different timing problems because both lived in its context and both used the word "timestamp." A human engineer reading the session log would have felt the drift — the agent was answering the wrong question using the vocabulary of the right one. But Archer had no mechanism for noticing that it was doing this. The context was structurally intact. It was logically stale.

When the engineer checked in at 3:00, Archer presented her with a confident patch. The patch modified the Redis lock TTL — the exact parameter that the dead-end hypothesis had already established was not the cause of the failure. The agent had done four hours of work and returned to its first wrong answer with the fluency and assurance of a model that had found strong local coherence in its own past reasoning. The 401 errors continued in production. The bug was not fixed. The session was not a token-window failure by any standard dashboard metric. It was a context rot failure: the agent had been defeated not by what it could not see, but by what it could not stop seeing.

### The Tetrahedron Read

This scenario is not primarily about code or authentication logic. It is about the four-way entanglement of **Structure**, **Logic**, **Implementation**, and **Outcome** that defines every context management decision in a long-running agent.

**Structure** is what the context window literally contains at any moment — which tokens occupy which positions, in what order, with what recency. In Archer's session, the structure of the context was a chronological stack: earliest reasoning at the top, most recent at the bottom. That structure is not neutral — high-salience tokens that repeat across dead-ends exert outsized influence on what the model generates next. The physical cause of this is explained in the next section; for now, note that Archer's context structure gave dead-end A a persistent vote it had not earned. Structure determines who gets to vote on the next token. Archer's structure gave dead-end A a permanent vote. Why that vote persists — and why it compounds — is the mechanism this chapter builds before evaluating any approach.

**Logic** is the inferential architecture that the agent builds on top of that structure — the chain of hypotheses, the deductions, the conditional branches. When the structure is contaminated by stale reasoning, the logic built on top of it inherits the contamination. Archer's clock-skew analysis was not illogical in isolation. It was illogical relative to the actual problem, because the logical steps were being shaped by premises — the lock vocabulary, the TTL framing — that had already been invalidated. Logic in an LLM agent is not a closed formal system. It is a probabilistic process that draws on everything in context. Stale structure produces distorted logic, and distorted logic produces confident wrong answers.

**Implementation** is the operational layer: which context management strategy Archer used, what its thresholds were, what it did when the window approached limits, and what it never did because no rule told it to. Archer used a passive strategy — what engineers call a sliding window with no active curation. Content entered the context chronologically and stayed there until the window limit forced truncation. There was no mechanism for Archer to mark a hypothesis as invalidated, compress dead-end reasoning into a compact tombstone, or expunge scaffolding code that had served its purpose. Implementation determines what the agent is physically capable of doing with its own history. Archer's implementation made context rot structurally inevitable in any session longer than its first hypothesis.

**Outcome** is what the engineer received at 3:00 PM: a wrong patch, delivered confidently, costing another half-day of lost debugging time and continued production errors. But the outcome extends further than a single wrong answer. Outcome in context management has a systemic dimension: if every long-running agent session in an organization silently degrades after the second hour, the organization learns — incorrectly — that agentic AI is unreliable at long tasks. The actual failure is architectural. The perceived failure is categorical. That misattribution is itself a cost, because it sends engineers toward the wrong remedies: better models, bigger windows, more careful prompting. None of those remedies address the implementation. The outcome loop closes not when the bug is fixed, but when the engineer correctly identifies where the leverage was.

The five approaches this chapter compares are five different answers to the same structural question: who controls context curation, by what mechanism, and at what point in the session? Each answer produces a characteristic failure mode, and naming those failure modes upfront is the only way to evaluate the approaches honestly. Passive truncation — no curation — fails by accumulation: dead-end tokens persist in the KV cache indefinitely, and context rot is structurally guaranteed in any session that outlives its first hypothesis. Compaction-on-threshold fails by semantic blindness: when the engine compacts on a size trigger, it compresses the oldest tokens, not the least relevant ones, which means it may discard a critical early constraint while preserving three hundred lines of invalidated scaffolding. RLM — the model writing code to process its own input — fails by problem mismatch: it is a batch-input architecture applied to a conversational working-memory problem, and it has no mechanism for updating its compression decisions as new information arrives mid-session. LCM — a deterministic DAG (directed acyclic graph: a structure with no cycles, so every compaction pass is guaranteed to terminate) enforcing lossless compaction — fails by excluding the model from curation decisions entirely: the engine knows what is structurally present but cannot know what is logically dead, and so its compaction is lossless in a technical sense while remaining semantically indiscriminate. ACM — the model using tools to actively manage its own context — fails when the model's metacognitive judgment is wrong, and it will sometimes be wrong. But this failure is categorically different from the others: it is visible in the tool calls the model makes, auditable in the session log, and correctable by a human or a subsequent agent pass. The other four failures are silent. That asymmetry is the architectural argument this chapter makes. Understanding why it holds requires understanding what context actually is — not as a metaphor, but as a mechanism.

## The Mechanism

When engineers talk about a model's "context," they typically mean something vague and metaphorical — a kind of working memory, a record of the conversation so far, the information the model "has access to." That framing is not wrong, but it is too abstract to be useful when diagnosing why an agent fails on hour four of a debugging session. To understand context rot, you need to understand what context physically is: a token sequence that populates a key-value cache, which in turn determines the attention computation that produces every subsequent token the model generates. There is no separate "memory system" sitting alongside the model that can be selectively updated. There is only the KV cache, and everything in it participates in every forward pass for as long as it persists.

The KV cache works as follows. Transformer models compute attention by projecting each token into three vectors: a query, a key, and a value. When generating token N+1, the model computes a dot product between the query of the current position and the keys of all prior positions, applies a softmax to produce attention weights — softmax is a normalization function: it converts the raw dot-product scores into weights that sum to 1.0, so stronger scores win a larger share of a fixed attention budget. This is what makes attention sink behavior concentrating rather than merely additive — high-salience tokens are not just influential, they are influential at the expense of everything else in the window — and then takes a weighted sum of the corresponding value vectors. This computation would be prohibitively expensive to redo from scratch on every generation step, so the infrastructure caches the key and value vectors for every prior token after they are computed. That cache is the KV cache. When Archer received its prompt at 9:47 AM and began reading the auth middleware, every token of that middleware — every function name, every variable, every comment — was encoded into a key-value pair and written into the cache. When Archer generated its first hypothesis about the race condition, those tokens joined the cache too. When it wrote test scaffolding, ran the tests, logged the results, and concluded the race condition was not the cause — all of that joined the cache. The cache does not have an "invalidated" flag. It does not have a recency decay. It has entries, and then it has more entries, and every entry participates in every forward pass until the session ends or the window is truncated.

This is why the token budget — the maximum sequence length the model can process — is a misleading proxy for session health. The token budget tells you when the KV cache is physically full; it tells you nothing about whether the cache's contents are logically coherent with the current state of the problem. Archer's session at 1:30 PM was at perhaps forty percent of its token budget. By any standard operational metric, the agent had plenty of room. But the cache held four hours of reasoning, a significant fraction of which was no longer true of the problem Archer was trying to solve. The dead-end race condition scaffolding — three hundred tokens of `lock_acquired_at` comparisons, Redis TTL assertions, and concurrency timing diagrams — was still present in the cache in full fidelity, generating attention weight on every forward pass, contributing to the probability distribution over every token Archer was about to produce. A session can be at forty percent of its token budget and be ninety percent corrupted by stale reasoning. Those two numbers are not related.

The specific mechanism that makes this corrupting rather than merely noisy is attention sink behavior. Not all tokens in the KV cache exert equal influence on the model's output. High-frequency tokens — punctuation, connectives — are widely distributed and tend to average out. But high-salience tokens — function names, error codes, repeated domain vocabulary — accumulate attention weight disproportionate to their position in the sequence. In Archer's case, the token `lock_acquired_at` appeared eleven times in the race condition scaffolding. The token `TTL` appeared eight times. The token `timestamp` appeared nineteen times across both the dead-end and the current hypothesis. Nineteen occurrences of `timestamp` in a context window pull the model's attention toward the lock-acquisition framing every time the current analysis touches anything time-related. This is not a reasoning error in the traditional sense. The model is not confused about what a timestamp is. It is being systematically biased by the statistical weight of a word that happens to appear in two different semantic contexts, one of which is no longer valid. Attention sinks do not distinguish between valid repetition and invalid repetition. They count occurrences and weight accordingly.

Context rot is therefore not a sudden failure. It is a gradient. In Archer's session, the drift was imperceptible at noon, noticeable to a careful reader by 2:00 PM, and catastrophic in the final patch delivered at 3:00. This gradient has a structural cause: as the session grows, the ratio of stale KV cache entries to current ones increases monotonically under any passive strategy, and the attention weight pulled toward stale vocabulary increases with it. The model is not getting dumber — its weights are unchanged, its architecture is unchanged — but the effective input to its reasoning is becoming less and less representative of the actual problem. The complication this introduces for system design is that the failure is self-concealing. A model generating tokens from a corrupted KV cache does not produce garbled output. It produces fluent, confident, internally coherent output that happens to be wrong. Archer's final patch was well-formatted, well-commented, and architecturally plausible. It was simply an answer to the question the cache remembered rather than the question the problem posed.

### The Tetrahedron Read

**Structure**, understood mechanistically, is the state of the KV cache at any given moment in the session. Every entry in that cache is a pair of vectors — a key vector encoding what that token is, and a value vector encoding what information it carries — written once at the moment the token is processed and never rewritten. Structure is append-only by architecture. In standard hosted inference APIs, there is no mechanism for modifying a KV cache entry mid-session — no mechanism for the model to reach back and zero out the keys and values corresponding to an invalidated hypothesis. This is not a design oversight; it is a consequence of how the attention computation achieves its efficiency. The cache is fast precisely because it is immutable. When Archer invalidated the race condition hypothesis, that invalidation was a semantic event — a conclusion expressed in tokens — but it produced no corresponding structural event in the cache. The keys and values that encoded the race condition scaffolding were still there, still participating, still exerting attention weight. Structure, in a transformer, has no concept of logical validity.

**Logic**, in an LLM agent, is an emergent property of the attention computation over the current KV cache state. This is the step that most architectural discussions elide. Archer did not have a reasoning module separate from its token generation. Its "logic" was the probability distribution over next tokens, conditioned on everything in the cache. When the cache contained predominantly clock-skew vocabulary but also a significant residue of lock-acquisition vocabulary, the probability distribution over Archer's next reasoning step was a mixture of both. The model did not "choose" to conflate the two problems. The query vectors at the current position produced high dot-product scores against both sets of key vectors, pulling the weighted sum of value vectors toward both framings simultaneously. Logic, for a transformer-based agent, is inseparable from cache state. You cannot have correct logic on top of a corrupted cache any more than you can have a correct calculation on top of a corrupted register. The substrate determines the computation.

**Implementation** is where the five approaches diverge, and the KV cache mechanism explains precisely why they diverge in the ways they do. A passive implementation writes to the cache and never curates it — context rot is guaranteed once dead-ends accumulate. A compaction-on-threshold implementation truncates oldest entries when a size limit is crossed — but oldest is not a proxy for least relevant, and the truncation does not selectively remove the high-salience attention sinks doing the most damage. RLM writes code to process the input before the session begins — but it cannot update that processing as the session evolves, because it has no mechanism for mid-session cache modification. LCM maintains a deterministic DAG of content and compacts it losslessly — but lossless compaction preserves stale tokens by definition, because it does not evaluate logical validity, only structural completeness. ACM instructs the model to use explicit tools — write to an external memory store, delete a context block, summarize a dead-end into a tombstone — which produces actual structural changes to what the next forward pass will attend to. Implementation is the only layer at which context rot can be architecturally addressed, not because it is the most important layer, but because it is the only layer with write access to what the cache contains.

**Outcome** for this mechanism section is not Archer's wrong patch — that belongs to the scenario. The outcome here is that the mechanism forces a binary architectural question: does the system give write access to cache contents to the engine, to the model, or to neither? Every approach in this chapter is an answer to that question. Passive says neither — content enters and persists without curation until the window truncates. Compaction-on-threshold says the engine — a size trigger fires and the engine compresses, using position as its proxy for relevance. RLM says the model, once, before the session begins — the model writes code to process its own input at initialization and has no further say. LCM says the engine, deterministically — a DAG enforces lossless compaction at every step, by structural rule, without semantic judgment. ACM says the model, continuously — the model uses explicit tools to add, compress, and delete context entries throughout the session. The mechanism does not tell you which answer is right. It tells you that not choosing is itself an answer. Archer had that answer, and it cost five hours and a wrong patch.

## Approach 1: Passive Truncation

The simplest thing a system can do with context is nothing. Tokens arrive, tokens are appended to the message history, the history is passed to the model on each call, and the cycle continues until the token budget is exhausted — at which point the oldest content is dropped from the front of the sequence to make room for the new. No curation. No evaluation of relevance. No distinction between a constraint that is still binding and a dead-end that has been superseded. The context grows forward and, when it must shrink, it shrinks from the beginning. That is passive truncation in its entirety, and it is the default behavior of every off-the-shelf agentic framework that does not explicitly configure something else. Engineers do not choose it so much as they inherit it by not choosing anything at all.

The pattern is worth looking at as code, because the code makes the architectural decision visible in a way that prose obscures:

```python
def run_agent(task: str, model: str, max_tokens: int) -> str:
    messages = []
    messages.append({"role": "user", "content": task})

    while True:
        response = call_model(model, messages)
        assistant_turn = {"role": "assistant", "content": response.content}
        messages.append(assistant_turn)

        if is_complete(response):
            return response.content

        next_input = execute_and_observe(response)
        messages.append({"role": "user", "content": next_input})

        # Truncate if over budget
        while token_count(messages) > max_tokens:
            messages.pop(0)
```

This is not a simplified illustration. This is the actual structure of a passive agent loop — the append, the check, the pop from the front. Notice what the loop does not contain: no function that evaluates the semantic relevance of any message before appending it, no flag that marks a turn as invalidated, no mechanism that compresses a dead-end into a summary before it persists. The only decision the loop makes about context content is positional: when space runs out, the oldest item goes. Every turn that Archer took between 9:47 AM and 3:00 PM was an iteration of a loop structurally identical to this one.

Why do engineers default to it? The answer is not negligence. It is that passive truncation works well enough under the conditions that most agents encounter during development and early deployment — conditions that are systematically different from production long-session operation. In development, sessions are short. A developer invoking an agent to generate a function or fix a one-step bug rarely accumulates more than a dozen turns. At that scale, the context window contains only valid, current reasoning, and the passive loop produces correct results reliably. The agent appears robust. The tests pass. The pattern gets promoted to production. What the developer has not observed — because the development session was too short to trigger it — is what happens when the session runs for four hours and makes three wrong turns before finding the right one. The token budget was never approached in testing. The truncation code never fired. The semantic corruption from dead-end accumulation never had time to compound. Development sessions are a biased sample, and passive truncation passes the sample while failing the population.

There is a second reason for the default, subtler than convenience: the token budget creates a metric that looks like a safety valve. Engineers who are aware of context management as a concern will often point to the budget as evidence that the system is managed. The window has a limit, the truncation fires when the limit is reached, and therefore the context cannot grow without bound. This reasoning is correct about the token count and wrong about the relevant failure mode. The token budget is a capacity constraint, not a quality constraint. It prevents the KV cache from growing infinitely; it does nothing to prevent the cache from filling with irrelevant content. A context window at eighty percent of its token budget and composed entirely of valid, current reasoning is healthier than a context window at forty percent of its budget composed of two dead-ends and one partial hypothesis. The budget number is visible on dashboards. The semantic coherence ratio is not. Engineers optimize for what they can measure, and passive truncation gives them a number that is measurable, meaningful-sounding, and orthogonal to the actual failure mode.

The causal chain from passive implementation to wrong output runs as follows, traced through Archer's specific session. At 11:15 AM, Archer appended the race condition scaffolding to its message history — approximately three hundred tokens of test code, log output, and intermediate conclusions, including eleven instances of `lock_acquired_at` and eight instances of `TTL`. At 11:45, Archer appended its conclusion that the race condition hypothesis was invalid. That conclusion was also a token sequence, also appended, but it did not — because the passive loop contains no such operation — remove, compress, or annotate the preceding scaffolding. The scaffolding and the invalidation now coexisted in the context: the evidence of a wrong path and the statement that it was wrong, side by side, equally persistent in the KV cache, equally available to every subsequent attention computation. When Archer began the clock-skew analysis at noon, its attention computation ran over a context that contained both the current hypothesis and the full vocabulary of the discarded one. The softmax over key-query dot products does not know that `lock_acquired_at` is semantically irrelevant to the clock-skew analysis. It knows that `lock_acquired_at` has a high-magnitude key vector that generates strong dot products with the query vectors of time-related tokens in the current analysis. Those strong dot products produce high attention weights. Those high attention weights shift the value-weighted sum that feeds the next-token probability distribution. The probability distribution shifts toward lock-acquisition vocabulary. Archer's next token is slightly more likely to be `lock` or `TTL` or `acquired` than it would be if the scaffolding had been absent. That slight shift compounds over hundreds of generation steps across two hours of clock-skew analysis. By 3:00 PM, the accumulated drift has steered the final patch back toward the lock TTL — not because Archer reasoned its way back there, but because the passive loop never gave it any other architectural option.

### The Tetrahedron Read

**Structure** in the passive approach is a monotonically growing chronological sequence with a hard truncation boundary at the front. Its defining characteristic is that it has no internal structure beyond position. There is no semantic layer, no validity layer, no recency-weighted layer. Every message appended to the history is a peer of every other message appended to the history: it occupies a position, it generates KV cache entries, and it participates in attention computations until it is truncated. The race condition scaffolding at position 40 in Archer's message history has exactly the same structural status as the clock-skew hypothesis at position 80. Structure cannot express the difference between "this is current reasoning" and "this was reasoning that I have since rejected." That distinction is a semantic property, and passive context management has no semantic layer. The structure is a flat tape.

**Logic** under passive truncation is therefore contaminated by design. This is the precise meaning of the phrase "fails by accumulation": it is not that the model accumulates too many tokens and runs out of room — it is that the model accumulates logically incompatible reasoning chains in a single undifferentiated context, and its attention computation produces a mixture distribution over all of them on every forward pass. For Archer, the logic of the clock-skew analysis was never operating over a clean context. It was operating over a context that contained the full residue of the race condition analysis, including all of its vocabulary, all of its intermediate conclusions, and — crucially — its original framing of the problem as a locking problem rather than a timing problem. The mixture is not averaged out. It is weighted by attention sink behavior, which means the highest-salience stale tokens exert influence disproportionate to their position in the sequence. Passive truncation does not produce noisy logic. It produces systematically biased logic, biased in the direction of whatever dead-end vocabulary is most salient in the cache.

**Implementation** of passive truncation requires no infrastructure beyond the append loop shown above. That is its engineering appeal and its architectural trap simultaneously. The zero-configuration cost makes it the path of least resistance for any team standing up an agent quickly. There are no design decisions to make about compaction thresholds, no DAG to maintain, no tool definitions to write for model-driven curation. The loop is ten lines. It handles short sessions correctly. It fails long sessions silently. The implementation gives engineers no signal when the failure mode activates, because the failure mode is not an error condition — it is a degradation of output quality that looks, from the outside, like correct operation. The agent keeps running. The API calls keep returning 200. The token budget stays within bounds. Everything looks fine until the engineer reads the final output and recognizes that the patch addresses the wrong parameter, at which point the passive loop has already completed and the session cannot be forensically decomposed to identify when the contamination began.

**Outcome** of passive truncation in production is not a single wrong answer — it is a performance envelope. Agents running passive truncation perform correctly on tasks whose natural session length falls within the window where context rot has not yet compounded: typically fewer than ten to fifteen turns on a focused task, or shorter on exploratory tasks where dead-ends accumulate faster. Outside that envelope, output quality degrades as a function of how many invalid hypotheses the session has generated and how much high-salience vocabulary they introduced into the cache. The envelope is invisible to standard monitoring because the signal — output correctness — requires a human evaluator or a ground-truth oracle to detect, neither of which is typically present in production agentic pipelines. What organizations observe instead is that their agents are "unreliable on long tasks" without being able to specify what length triggers the failure or why. That observational fuzziness is the outcome of choosing an implementation with no diagnostic surface. Passive truncation fails by accumulation; it is also diagnosed by accumulation — of confused engineers, wrong outputs, and misattributed root causes. The obvious engineering response to this diagnosis is to add a size trigger: let the engine watch the token budget and compact when a threshold is crossed. That response correctly identifies the symptom — accumulation — and misidentifies the cure. Approach 2 shows why.

## Approach 2: Compaction-on-Threshold

Compaction-on-threshold is passive truncation with a conscience. It watches the token count, and when the count crosses a configured threshold — say, seventy percent of the maximum budget — it fires a summarization call: a separate model invocation that takes the oldest N messages, compresses them into a summary block, and replaces those messages with the summary before the session continues. The total token count drops. The session proceeds. From a monitoring dashboard, the pattern looks like active management: the budget graph rises, hits the threshold, drops sharply at compaction, rises again. Engineers can see the system responding to pressure. This is a genuine improvement over passive truncation in one respect — the context does not grow without bound — and it is almost no improvement at all in another, which is the respect that matters.

The pattern's mechanical structure makes the problem visible:

```python
def run_agent_with_compaction(
    task: str,
    model: str,
    max_tokens: int,
    compaction_threshold: float = 0.70,
    compaction_window: int = 10,
) -> str:
    messages = []
    messages.append({"role": "user", "content": task})

    while True:
        response = call_model(model, messages)
        messages.append({"role": "assistant", "content": response.content})

        if is_complete(response):
            return response.content

        next_input = execute_and_observe(response)
        messages.append({"role": "user", "content": next_input})

        # Compact when threshold is crossed
        if token_count(messages) > max_tokens * compaction_threshold:
            oldest = messages[:compaction_window]
            summary = summarize(model, oldest)          # one model call
            messages = [{"role": "user", "content": f"[COMPACTED SUMMARY]: {summary}"}] + messages[compaction_window:]
    
```

The compaction logic is four lines. And those four lines contain the full failure mode: `messages[:compaction_window]` selects the oldest messages. Not the least relevant messages. Not the most redundant messages. Not the messages whose hypotheses have been invalidated. The oldest messages. Position is the only criterion, because position is the only property the engine can evaluate without understanding the content. Changing the selection criterion to "least semantically relevant" would require the engine to evaluate content — which requires calling the model — which changes the architecture from engine-driven to model-participatory. That is not a tweak to this approach; it is a different approach. The design decision embedded in `messages[:compaction_window]` is not laziness — it is a structural constraint of engine-driven curation. The engine does not know whether the oldest block contains a live constraint or a dead-end. It knows the block is old, and it compresses it.

Why do engineers adopt this pattern? Because it is a real, observable response to a real, observable problem. A system that compacts is visibly different from a system that does not — the budget graph proves it, the logs show the summarization calls, the session continues past the point where a passive system would have begun truncating critical content from the front. Compaction-on-threshold also has a strong intuitive appeal: human experts managing long tasks do something that looks similar, reviewing what has been done, summarizing progress, and moving forward with a shorter working memory. The analogy feels right. What the analogy misses is that a human expert reviewing their own work applies semantic judgment when deciding what to summarize and what to keep verbatim. A compaction algorithm that selects by position applies no judgment at all. The human asks: what does the next step of this task actually need? The threshold trigger asks: what is oldest?

In Archer's session, the compaction-on-threshold failure is not abstract. At 9:47 AM, the first messages Archer processed were the task description and the initial codebase read: the engineer's instruction specifying that the bug occurred in roughly one session in forty, always under load, never in staging, and the auth middleware source. That content — the failure rate, the load dependency, the staging immunity — was the most important information in the entire session. It was the constraint that ruled out deterministic bugs, ruled out code-path errors that would reproduce consistently, and narrowed the search space to race conditions, timing problems, and environment-specific configuration differences. It was also the oldest content in the context. When the token budget crossed the compaction threshold sometime around 1:00 PM, the engine selected the oldest block — the task description and the initial middleware read — and passed it to the summarizer. The summarizer, operating on a short context window with a compression objective, reduced "one session in forty, always under load, never in staging" to something in the register of "intermittent authentication errors under certain conditions." The specificity that distinguished this bug from the entire class of deterministic auth failures was gone. The summary was not wrong — intermittent auth errors under certain conditions is an accurate description — but it was no longer precise enough to constrain the search. Archer, resuming its clock-skew analysis after compaction, was now reasoning over a context that preserved three hundred tokens of invalidated race condition scaffolding and had lost the sentence that made the race condition hypothesis implausible in the first place.

The causal chain from compaction trigger to wrong output runs as follows. The token count crosses seventy percent of budget. The engine selects the oldest ten messages: the task specification and the initial codebase read. The summarizer compresses them into a single system message. The compression loses the load-dependency and staging-immunity constraints — not through error, but through the fundamental mechanics of summarization, which trades specificity for brevity. The compressed summary re-enters the context as a system message at position zero. The subsequent context contains the invalidated race condition scaffolding at positions eleven through thirty, the invalidation statement at position thirty-one, and the clock-skew analysis from positions thirty-two onward — none of which contains the load-dependency constraint, because that constraint lived in the block that was just summarized away. When Archer generates its next hypothesis, it is reasoning over a search space that has quietly expanded: without the load-dependency constraint, a deterministic bug in the clock synchronization path is not ruled out. More hypotheses are now plausible. Archer pursues them. Each new dead-end adds its vocabulary to the cache. The session extends. The threshold fires again. The next compaction selects the summarized system message — compressing a compression — and the invalidated race condition scaffolding that has been doing the most damage survives untouched because it is not the oldest content. By the third compaction cycle, the context is a palimpsest: multiple layers of lossy summarization at the top, full-fidelity dead-end scaffolding in the middle, and a thin slice of current analysis at the bottom. The most important information in the session has been compressed three times. The most damaging information has never been touched.

### The Tetrahedron Read

**Structure** under compaction-on-threshold is a segmented sequence: one or more summary blocks at the leading edge, followed by the uncompacted tail of the session. Each compaction cycle adds another summary layer and advances the boundary of the uncompacted tail. The structure is no longer purely chronological — the summary blocks are positionally oldest but informationally lossy, while the uncompacted tail is positionally recent but semantically mixed. This segmentation creates a specific structural hazard that passive truncation does not produce: the most critical early constraints, which were stated once at session initialization and never repeated, are precisely the content most likely to be compacted first. High-salience tokens that appear repeatedly — `lock_acquired_at`, `TTL`, `timestamp` — are distributed across the uncompacted tail and survive every compaction cycle. Low-salience tokens that appear once — "one session in forty," "always under load," "never in staging" — live in the oldest messages and are compressed on the first trigger. Structure under compaction-on-threshold systematically favors frequency over importance, and frequency is not a proxy for relevance.

**Logic** under compaction-on-threshold degrades through a different mechanism than under passive truncation, and the difference is worth naming precisely. Passive truncation produces logic contaminated by the full fidelity residue of dead-end reasoning. Compaction-on-threshold produces logic contaminated by the lossy summary of critical constraints combined with the full-fidelity residue of dead-end reasoning — which is strictly worse, because the agent now has both the wrong vocabulary amplified by attention sinks and the wrong search space produced by constraint loss. After the first compaction in Archer's session, the clock-skew analysis was no longer constrained by the load-dependency and staging-immunity specifications that made certain classes of bug implausible. The agent's probability distribution over next hypotheses was broader, not narrower, than it would have been under the passive approach — because passive truncation at least preserved the original task description, even as it failed to remove the dead-end scaffolding. Compaction-on-threshold removed the constraint and kept the noise. The logic it produced was contaminated in both directions simultaneously.

**Implementation** of compaction-on-threshold is more complex than the passive loop and significantly less complex than the approaches that follow. It requires a threshold parameter, a window-size parameter, and a summarization call — three design decisions, each of which introduces a new failure mode. A threshold set too high fires too late, after the context has already rotted. A threshold set too low fires too frequently, producing cascading lossy summarizations that compound constraint loss. A window size set too large compresses too much critical content per cycle. A window size set too small leaves the high-salience dead-end scaffolding intact through multiple cycles. The summarization call quality determines how much specificity is preserved — and no summarizer, however capable, can preserve specificity it has not been told to prioritize, because it does not know which specifics are constraints and which are elaborations. The implementation appears to give engineers control through parameters; it actually gives them three knobs whose optimal settings are task-dependent, session-length-dependent, and unknowable in advance.

**Outcome** of compaction-on-threshold in production is the same wrong answer as passive truncation, reached by a different path, with a monitoring dashboard that makes the failure harder to diagnose. Under passive truncation, the session log is a complete chronological record; a forensic analyst can read it and identify exactly when the dead-end vocabulary began contaminating the clock-skew analysis. Under compaction-on-threshold, the session log contains summary blocks that have replaced the original content — the original task specification is gone, replaced by a lossy compression that does not record what was lost. The failure is not only harder to prevent; it is harder to reconstruct. Engineers examining a compaction-on-threshold failure often conclude that the summarizer performed poorly, adjust the summarization prompt, and redeploy — which addresses the quality of the compression without addressing the fact that position-based selection compresses the wrong content regardless of summarizer quality. The real fix is not a better summarizer. It is a different criterion for selecting what to compact. And that criterion requires semantic judgment — which the engine does not have, but the model does. Recognizing this is the insight that motivates the next two approaches, each of which attempts, in different ways, to give the model a role in deciding what the context should contain. The first of those attempts — RLM — does so before the session begins, in a single batch operation, and discovers a problem that the compaction-on-threshold approach does not have: the problem of a session that changes its own nature as it runs.

## Approach 3: RLM — Recursive Language Models

The first two approaches share a common assumption: that context management is a problem of what to keep. Passive truncation keeps everything until forced to drop the oldest. Compaction-on-threshold keeps everything recent and compresses everything old. Both treat the context as a sequence to be curated by position. RLM — the Recursive Language Model architecture introduced by Zhang, Kraska, and Khattab (2026) — breaks from this framing entirely. It treats context management as a problem of what to compute. Rather than deciding which tokens to preserve, it gives the model a programming environment and asks it to write the code that extracts what it needs from the input. The model does not receive the input directly. It receives metadata describing the input, and it writes Python to process the input programmatically — chunking it, querying it, spawning recursive sub-calls via `llm.query()` on the pieces it cares about — until it has assembled the information required to produce a final answer. The loop terminates when the model sets a `Final` variable in the REPL environment. This is a genuinely different architectural idea, and it works remarkably well on the class of problems it was designed for. It also fails completely on the class of problems that agentic debugging sessions represent. Understanding why requires understanding what the architecture actually does, rather than what it superficially resembles.

The REPL loop is the architecture's core mechanism. On each iteration, the model receives a short history of prior REPL outputs — not the full input, but the captured stdout of prior code executions — appended to the original metadata. It generates the next block of Python code. The code executes. Stdout is captured and appended to the short history. The model generates the next code block. This continues until `Final` is set. The architecture's insight is that this loop can process inputs orders of magnitude larger than the context window, because at no point does the full input occupy the context: only the metadata, the short REPL history, and the current code block are ever present simultaneously. The model's effective reach over the input is unbounded by the context window, mediated instead by the quality of the code it writes to traverse the input programmatically.

```python
# Simplified RLM REPL loop (Zhang, Kraska, Khattab 2026)
def run_rlm(input_document: str, task: str, model: str) -> str:
    metadata = extract_metadata(input_document)   # structure, length, chunk map
    repl_history = []
    local_env = {"document": input_document, "llm": LLMProxy(model)}
    Final = None

    while Final is None:
        prompt = build_prompt(task, metadata, repl_history)
        code = call_model(model, prompt)           # model writes Python

        stdout, local_env = execute_in_repl(code, local_env)
        repl_history.append({"code": code, "stdout": stdout})

        Final = local_env.get("Final")             # set by model when done

    return Final
```

The key line is `local_env = {"document": input_document, "llm": LLMProxy(model)}`. The input is placed into the REPL environment as a variable. The model never reads it directly as a prompt — it writes code to read it. `llm.query(snippet)` spawns a sub-call on a chunk of the document, recursively. The architecture's strength on benchmarks like S-NIAH, OOLONG, and BrowseComp — tasks that require locating specific information within inputs two orders of magnitude beyond the context window — follows directly from this structure. If the task is "find the authentication token buried in a 2-million-token codebase," RLM can write a chunking loop, query each chunk, and surface the token without ever placing the full codebase in context. That is a real and significant capability.

Now apply RLM to Archer's session. At 9:47 AM, the engineer's instruction arrives. In the RLM architecture, Archer would receive metadata describing the auth codebase — file count, module structure, dependency graph, line count — and a task description. It would write Python to traverse the codebase: open the auth middleware, chunk it, call `llm.query()` on the token-refresh handler, extract the candidate failure paths, synthesize a diagnosis. All of this is within RLM's design envelope. The problem begins at 11:15 AM, when the race condition hypothesis collapses. In a conversational agentic session, the next step is to take the result of the test — the stdout, the timing measurements, the invalidation conclusion — and incorporate it into the working context as a new premise that constrains the next hypothesis. RLM has no mechanism for this. The REPL loop is designed to process a fixed input document that exists at session initialization. The four-hour debugging investigation is not a document. It is a conversation that generates new information on every turn — test results, tool outputs, intermediate conclusions — information that did not exist when the session began and that changes the meaning of everything that came before it. RLM assumes the context problem is: the input is too large to fit. The actual problem in Archer's session is: the context fills with self-generated content that accumulates logical incompatibilities over time. These are different problems. RLM solves the first one precisely and has no answer to the second.

The failure mode is not a weakness in execution. RLM executes its design correctly. The failure is a category error in problem framing. The compaction-on-threshold approach misidentifies the cure for context rot; RLM misidentifies the disease. It arrives at the debugging task having already answered a different question — "how do I process an input larger than my context window?" — and finds that the answer is inapplicable. The session it encounters does not have a large fixed input waiting to be processed. It has a small initial input and a growing sequence of self-generated turns, each of which is produced by the model's own prior reasoning and each of which can invalidate premises established earlier in the session. No code that the model writes at session initialization can account for content that does not exist until turn thirty. The REPL loop terminates when `Final` is set; there is no architectural slot in which the model can revise its decomposition strategy mid-session in response to an unexpected test result. RLM is a batch processor applied to a streaming problem, and the mismatch is architectural, not parametric.

### The Tetrahedron Read

**Structure** in the RLM architecture is not a message history at all — it is a REPL environment plus a short stdout log. This is the approach's most radical structural departure from the other four. Where passive truncation, compaction-on-threshold, LCM, and ACM all manage some form of conversation thread, RLM replaces the thread with a program execution trace. The input document lives in a variable; the model's reasoning lives in code; the history is a sequence of code-stdout pairs, not assistant-user exchanges. This structure is well-suited to the task of navigating a large fixed corpus: the model can write arbitrarily complex traversal logic, and the REPL environment gives it access to the full document without placing the full document in context. The structure is unsuited to the task of maintaining a conversational working memory across a multi-hour session, because a conversational working memory is not a document that can be placed in a variable. It is a sequence of events, each of which is contingent on the ones before it and constraining of the ones after it. The REPL environment has no representation for contingency. It has variables, functions, and stdout. When Archer invalidated the race condition hypothesis at 11:45 AM, that invalidation was a contingent event — it changed the logical status of everything the session had established so far. RLM's structure cannot encode that change. The REPL environment either does or does not contain the invalidation as a string in a variable; it cannot represent the fact that the invalidation changes the relevance of prior content.

**Logic** in the RLM architecture is expressed as code rather than as natural language reasoning chains. This is genuinely powerful for certain problem classes: code is precise, executable, and inspectable in a way that natural language is not. When the task is to locate a specific fact in a large document, code-expressed logic — `for chunk in chunks: if matches_criterion(chunk): llm.query(chunk)` — is more reliable than natural language reasoning over a long context, because the traversal is deterministic and the model's attention is focused on small, well-defined pieces of the input. When the task is to debug a complex intermittent failure over a multi-hour session, code-expressed logic is the wrong substrate for the reasoning that needs to happen. The reasoning Archer needs to do — "the race condition hypothesis is now invalidated; what constraints does this place on the remaining hypothesis space?" — is inferential and contingent, not procedural and deterministic. RLM requires the model to be a competent programmer on every iteration, generating correct Python from scratch against a REPL environment that accumulates state across iterations. That requirement holds in the document-processing context where the environment is stable. It becomes an additional cognitive burden in a debugging session where the environment is being modified by external tool calls — test runners, log scrapers, API probes — whose outputs are not predictable at session initialization.

**Implementation** of RLM requires infrastructure that none of the other four approaches need: a REPL execution environment, a metadata extraction layer, an `LLMProxy` that mediates recursive sub-calls, and a mechanism for capturing and appending stdout to the model's short history. This is a non-trivial build. The benchmarks on which RLM demonstrates strong performance — S-NIAH, OOLONG, BrowseComp — were constructed to evaluate exactly the capability the architecture provides: navigating large fixed inputs with precision. Those benchmarks are not conversational. They present the model with a document and a question; they do not present the model with a task that requires it to update its world model in response to information generated by its own prior actions. An engineering team that adopts RLM for long-session agentic work on the basis of those benchmark scores is generalizing from the right result on the wrong test. These benchmarks were not independently selected to stress-test RLM — they were chosen to demonstrate the capability the architecture provides. This is standard practice in architecture papers and a standard reason to treat benchmark performance as evidence of fit-for-purpose, not general superiority. The implementation is correct; the test is not representative of the deployment environment. Strong benchmark performance on synthetic needle-in-haystack tasks does not transfer to conversational working memory management, because the tasks are not in the same problem class.

**Outcome** of applying RLM to Archer's session is not context rot — it is task abandonment. RLM cannot produce a five-hour debugging session, because it has no mechanism for the conversational turns that constitute a five-hour debugging session. It produces a single-shot analysis of the initial codebase, correctly structured and likely more thorough than what Archer's passive implementation generated in its first hour. Then the session ends, because the REPL loop has set `Final` and returned. When the race condition hypothesis needs to be tested, the test results need to be incorporated, and the clock-skew analysis needs to begin, RLM has no loop iteration in which those events can occur. The outcome is not a wrong answer; it is a category of answer that does not match the task. This is, in one sense, a cleaner failure than passive truncation or compaction-on-threshold — at least the failure is observable, because the session terminates rather than continuing to produce plausible-but-wrong output. But observability does not make the failure useful. Archer's engineer needed a five-hour debugging partner, not a one-shot codebase analyst. RLM's failure by problem mismatch reveals what the right problem actually is: not "how does the agent process a large fixed input?" but "how does the agent maintain a coherent, curated working memory across a session whose content is generated by the session itself?" The answer to that question requires an architecture that was designed for it — one that tracks the structure of the session as a graph of dependencies rather than a sequence of positions, and that enforces compaction rules deterministically as the session evolves. That is the problem LCM was built to solve.

## Approach 4: LCM — Lossless Context Management

LCM — introduced by Ehrlich and Blackman (2026) — is built on a recognition that the previous two engine-driven approaches share a structural flaw: they destroy information. Compaction-on-threshold summarizes the oldest block and discards the original. Passive truncation drops the oldest tokens outright. In both cases, what was said early in the session becomes unrecoverable once the compaction or truncation fires. LCM's foundational architectural move is to separate the concern of what is stored from the concern of what the model sees. It does this with two pillars: an Immutable Store, where every message is persisted verbatim and never modified, and an Active Context, which is the window actually sent to the model on each turn, assembled dynamically from recent messages plus summary nodes that represent compacted history. The model always sees a context that fits within its token budget. The full session history is always recoverable. These two properties hold simultaneously, and in their combination LCM represents a genuine architectural advance over the approaches that preceded it.

The core data structure is a hierarchical directed acyclic graph (DAG) maintained in the Immutable Store — Ehrlich and Blackman's reference implementation uses embedded PostgreSQL. Each node in the DAG is either a raw message node, written once when a turn completes and never modified, or a summary node, written by a compaction pass that consumed one or more raw message nodes and produced a shorter representation of their content. Summary nodes can themselves be summarized, producing a multi-resolution hierarchy in which the earliest session content is represented at the highest level of abstraction and the most recent content is represented at full fidelity. The Active Context is assembled by the engine on each turn by walking the DAG: recent raw nodes are included verbatim, older nodes are included as their summary representations, and the total token count is kept within budget. When the engineer issues a query about something that was summarized away, `lcm_expand` recovers the original raw nodes from the Immutable Store — the originals are never gone, only deprioritized in the Active Context.

```
# LCM DAG structure — Archer session at 3:00 PM
#
# IMMUTABLE STORE (PostgreSQL — all nodes, never modified)
# ─────────────────────────────────────────────────────────
#
#   [S2: Summary of S1 + msg_006..010]          ← compaction pass 2
#         │
#   [S1: Summary of msg_001..005]               ← compaction pass 1
#         │
#   [msg_001: task description + constraints]   ← "1-in-40, under load"
#   [msg_002: middleware read]
#   [msg_003: race condition hypothesis]
#   [msg_004: scaffolding + test output]        ← lock_acquired_at ×11
#   [msg_005: invalidation of hypothesis]
#   [msg_006..010: early clock-skew analysis]
#   [msg_011..020: continued clock-skew analysis]
#   [msg_021..025: current turn]                ← most recent, full fidelity
#
# ACTIVE CONTEXT (assembled by engine, sent to model each turn)
# ─────────────────────────────────────────────────────────────
#
#   [S2] + [msg_011..020] + [msg_021..025]
#
#   S2 is a lossy compression of everything from msg_001 through msg_010,
#   including the task constraints AND the race condition scaffolding.
#   msg_011 through msg_025 are present in full fidelity,
#   including whatever clock-skew dead-ends have accumulated since.
```

![Figure 4 — LCM DAG Structure: Immutable Store vs Active Context](figures/figure4_lcm_dag.png)
*Figure 4: LCM separates what is stored from what the model sees. The separation solves recoverability. It does not solve relevance.*

Compaction-on-threshold has an implicit assumption: that the summarizer will return output shorter than its source. A verbose model, or a source block that is already a summary node, can violate this — the session crosses the threshold, fires a compaction call, and the resulting context is still over budget. LCM addresses this with a three-level escalation guaranteed to converge regardless of summarizer behavior. The compaction logic has a three-level escalation that guarantees convergence — a property the previous approaches lack. When the Active Context token count crosses the soft threshold, the engine triggers an asynchronous compaction between turns: the oldest raw message block is passed to the LLM summarizer, a detail-preserving summary node is written to the DAG, and the raw messages are removed from the Active Context but preserved in the Immutable Store. If the resulting Active Context still exceeds the hard threshold — because the summarizer produced output longer than expected, or because the source block was already a summary node — the engine fires a second compaction pass at bullet-point compression, targeting half the previous output size. If the context still exceeds budget after two LLM-based passes, the engine applies deterministic truncation without a model call, which is guaranteed to converge. No summarization pass can produce output longer than its source; the three-level escalation ensures the Active Context always fits within budget before the next model call fires. This convergence guarantee is LCM's most important engineering property, and it is one that neither passive truncation nor compaction-on-threshold provides: passive truncation is deterministic but unintelligent; compaction-on-threshold is intelligent but not guaranteed to converge on a single summarization pass.

These are genuine strengths, and they should be credited before the failure mode is named. Below the soft threshold, LCM has zero cost: messages flow through unchanged, no compaction calls fire, and the Immutable Store accumulates a perfect provenance record at no latency penalty. Above the threshold, convergence is guaranteed by escalation rather than by hoping the summarizer performs within spec. And at any point in the session, `lcm_expand` can recover the full original text of any compacted node — which means a human reviewer can always reconstruct exactly what the model was told, even after multiple compaction passes. No other engine-driven approach provides all three of these properties simultaneously. LCM is, within its design space, a well-engineered system.

The design space is the limit. LCM correctly separates the Immutable Store from the Active Context, and it solves the convergence and recoverability problems that engine-driven compaction leaves open. What it does not solve — and what its architecture structurally cannot solve — is the problem of which content belongs in the Active Context at each step of the session. The engine assembles the Active Context by walking the DAG and selecting the most recent raw nodes plus summary representations of older content. Recent is the selection criterion. In Archer's session, this means the race condition scaffolding — three hundred tokens of `lock_acquired_at` comparisons written at 11:15 AM, invalidated at 11:45 AM, never touched by any compaction pass because it sat in the middle of the message history rather than at the oldest end — remained in the Active Context in full fidelity for the entire afternoon. The Immutable Store had a perfect record of it. The Active Context had the same perfect record of it. The model attended to it on every forward pass, with the same attention sink dynamics described in Section 2, for the same reason as in the passive and compaction-on-threshold cases: the engine selected content by recency, and the race condition scaffolding was not the oldest content. LCM's three-level escalation, its PostgreSQL-backed DAG, its `lcm_expand` recoverability — none of these properties address the question of whether a given block of content is logically current or logically dead. They address the question of whether the content is recoverable. Recoverability and relevance are orthogonal properties. LCM optimizes the first and does not touch the second.

Apply this to Archer's session at the moment the first compaction fires — call it 12:30 PM, when the Active Context crosses the soft threshold after two and a half hours of work. The engine walks the DAG and selects the oldest block for compaction: `msg_001` through `msg_005`, which contain the original task description with its load-dependency and staging-immunity constraints, the initial middleware read, the race condition hypothesis, the scaffolding, and the invalidation statement. The summarizer compresses all five into a summary node `S1`. The constraints — "one session in forty, always under load, never in staging" — survive the summarization at some level of fidelity, probably reduced in specificity as the summarizer balances breadth and length. The race condition scaffolding also survives the summarization at some level of fidelity. But `msg_006` through `msg_025` — which include the clock-skew analysis in full detail, including whatever dead-ends it has already produced, including every instance of `lock_acquired_at` vocabulary that leaked from the race condition framing into the timing analysis — remain in the Active Context untouched, because they are not the oldest content. The compaction pass improved the situation slightly: the task constraints are preserved in `S1` rather than dropped as in raw truncation, and they are recoverable in full via `lcm_expand` if someone thinks to ask. The race condition scaffolding's influence on the Active Context is unchanged. The significant infrastructure investment — PostgreSQL, DAG, provenance tracking, three-level escalation — exists to compensate for the engine making temporal decisions without semantic understanding, and it compensates for the recoverability problem precisely while leaving the relevance problem intact.

### The Tetrahedron Read

**Structure** in LCM is the richest of the five approaches: a hierarchical DAG in a persistent store, with raw nodes at the leaves, summary nodes at interior levels, and the Active Context assembled as a view over the DAG rather than as the DAG itself. This structural sophistication produces the recoverability guarantee that distinguishes LCM from all engine-driven approaches that modify content in place. The structure also makes the failure mode precise in a way that the simpler approaches do not. In passive truncation and compaction-on-threshold, the position-based selection criterion is an implementation decision that could, in principle, be replaced with a different criterion. In LCM, the position-based criterion is embedded in the DAG traversal logic: the engine walks the DAG from oldest to newest, selects the oldest compactable block, and fires the summarizer on it. Changing the selection criterion would require changing the DAG traversal logic — which would require the traversal to evaluate semantic properties of nodes, which the engine cannot do. The structure is sophisticated enough to make the limitation visible: LCM's DAG can represent the history of the session in full resolution, but it cannot represent the logical status of any node within that history. The DAG knows that `msg_004` contains the race condition scaffolding. It does not know that `msg_004` is logically dead.

**Logic** under LCM is shaped by the Active Context, which is shaped by the DAG traversal. Because the traversal is deterministic and position-based, the logic the model applies on each turn is a function of recency, not of relevance. This produces the same contamination dynamic as in the previous engine-driven approaches, with one important difference: LCM's multi-resolution hierarchy means the contamination has a structured form. The oldest content is present as a high-abstraction summary; the middle content is present as lower-abstraction summaries or raw nodes depending on compaction timing; the most recent content is present in full fidelity. The model's attention is therefore distributed across three levels of representation simultaneously — compressed constraints at the top, partially-compressed intermediate reasoning in the middle, full-fidelity recent content at the bottom. In Archer's session, the attention sink dynamics operate most strongly on the full-fidelity recent content, because those nodes have the highest token count and the highest key-vector magnitude for domain-specific vocabulary. The dead-end scaffolding in the middle of the session history, if it has not yet been compacted, exerts full-fidelity attention sink influence. If it has been compacted into a summary node, its influence is reduced but not eliminated — the summary node still contains the vocabulary of the dead-end, simply in smaller quantity. LCM reduces the severity of context rot compared to passive truncation; it does not eliminate it, because elimination would require the system to distinguish logically dead content from logically current content, and that distinction is not a structural property of the DAG.

**Implementation** of LCM is the most complex of the five approaches by a significant margin. The Immutable Store requires a persistent database — the reference implementation uses embedded PostgreSQL, which means the system carries a relational database as a dependency of its context management layer. The DAG requires provenance tracking: every summary node must maintain references to the raw nodes it was derived from, so that `lcm_expand` can reconstruct the originals. The three-level escalation requires the engine to execute up to three compaction calls per threshold crossing, monitor the token count after each pass, and fall back to deterministic truncation if the LLM-based passes do not converge. The soft and hard thresholds must be tuned per deployment, and their interaction — soft threshold triggers async compaction, hard threshold triggers blocking compaction — introduces latency characteristics that must be managed in production. All of this infrastructure exists to solve the recoverability problem and the convergence problem, both of which are genuine problems worth solving. The implementation cost is justified by those solutions. It does not purchase a solution to the relevance problem, which remains open regardless of how well the infrastructure performs. A team that builds LCM's full infrastructure and then discovers that the Active Context still fills with semantically stale content has paid a significant engineering cost for recoverability without purchasing the semantic curation that the task required.

**Outcome** of LCM in Archer's session is better than the outcome of the previous three approaches in one measurable dimension — the original task constraints are recoverable after compaction — and equivalent in the dimension that determines the final answer: the Active Context is still contaminated by the race condition scaffolding, and the model still produces a patch that addresses the wrong parameter. The `lcm_expand` capability is genuinely useful for post-hoc forensics: a reviewer who suspects the agent lost a critical constraint can retrieve `msg_001` from the Immutable Store and verify that the load-dependency specification was present at session initialization. That is valuable for debugging the agent's failure, not for preventing it. LCM correctly identifies that the Immutable Store and the Active Context should be separate concerns — this is the architectural insight that distinguishes it from the approaches that modify or discard content in place. The question it does not answer is who decides what belongs in the Active Context at each step of the session. The engine decides, by DAG traversal order, by temporal position, without semantic judgment. The model — which has semantic judgment — is excluded from that decision. ACM is the approach that answers the question LCM leaves open, by giving the model the tools to make that decision continuously, throughout the session, with write access to the content of the Active Context rather than merely write access to the tokens it generates next.

## Approach 5: ACM — Agentic Context Management

Every approach examined so far has treated context curation as a problem for the system to solve on the model's behalf. The engine watches token counts, fires summarization calls, walks DAG traversals, applies escalation hierarchies — and the model generates its next token into whatever Active Context the engine has assembled. The model is the beneficiary of the curation, not a participant in it. ACM — Agentic Context Management — is the approach that abandons this arrangement. Its architectural premise is that the entity best positioned to judge what belongs in the working context of a long-running session is the entity doing the work: the model itself. ACM gives the model four tools and the authority to use them at any point in the session, continuously, in response to its own evolving understanding of what the task requires. The engine does not decide when to compact, what to compact, or what to preserve. The model does. The engine's job is to maintain the infrastructure that makes those decisions safe to execute: an immutable conversation log that records everything, and a context view that reflects the model's accumulated curation decisions without ever destroying the underlying record.

The two-structure design is ACM's foundational architectural commitment. The Conversation Log is append-only: every message, every tool call, every tool result is written as newline-delimited JSON with a sequential identifier — `msg-001`, `msg-002`, and so on — and never modified after write. The log is the ground truth of the session. The Context View is a computed projection of the log: assembled fresh on every turn by replaying the accumulated context management directives the model has issued so far, applied against the full log. The Context View is ephemeral. It can change between turns without touching the log. When the model calls `remove_context` on `msg-004` through `msg-007`, those messages disappear from the next Context View. They remain in the log, retrievable by ID or by the label the model assigned when removing them. The separation between log and view is what makes ACM's failure mode recoverable in a way that no prior approach achieves: nothing is ever destroyed, only projected out of the current working window.

The four tools define the full scope of what the model can do with its own context:

```python
# ACM tool definitions — available to the model as callable functions

def remove_context(
    message_ids: list[str],
    summary: str | None = None,     # optional tombstone replacing removed messages
    label: str | None = None,       # optional label for later retrieval
) -> None:
    """
    Remove specified messages from the Context View.
    Originals remain in the Conversation Log, unmodified.
    If summary provided, a summary node replaces the removed block in the View.
    If label provided, retrieve_context(label) restores the originals.
    """

def pin(message_ids: list[str]) -> None:
    """
    Mark messages as undeletable in the Context View.
    Pinned messages bypass ALL removal mechanisms, including
    future remove_context calls and any engine-driven compaction.
    """

def unpin(message_ids: list[str]) -> None:
    """
    Remove protection from pinned messages, making them
    eligible for future remove_context calls.
    """

def retrieve_context(
    label: str | None = None,
    message_ids: list[str] | None = None,
) -> list[Message]:
    """
    Pull previously removed content back into the Context View.
    Retrieval by label (if assigned at removal) or by specific message IDs.
    Retrieved messages re-enter the View at current position.
    """
```

These four tools are the complete interface. There is no threshold parameter, no compaction window, no DAG traversal — the model decides when to act, what to act on, and what to preserve as a tombstone summary. The infrastructure's only job is to ensure that every removal is reversible and every pinned message is inviolable. The model's job is to exercise semantic judgment about its own working memory, turn by turn, as the session evolves.

Apply this to Archer's session from 9:47 AM forward, and the architectural difference from the previous four approaches becomes concrete. At session initialization, before any hypothesis is formed, Archer reads the task description: one session in forty, always under load, never in staging. In an ACM-equipped session, Archer immediately recognizes these three constraints as the session's most critical information — the specifics that distinguish this bug from the entire class of deterministic auth failures and that will constrain every hypothesis for the duration of the session. Archer calls `pin(["msg-001"])`. The task description is now undeletable. No subsequent `remove_context` call can touch it. No engine-driven compaction can touch it. For the rest of the session, those three constraints are guaranteed to be present in every Context View, regardless of how many turns accumulate, regardless of how many compaction cycles fire, regardless of how large the session grows.

At 11:45 AM, the race condition hypothesis is invalidated. In the passive, compaction-on-threshold, and LCM sessions, Archer notes the invalidation in its working log and continues — the scaffolding persists in the Context View, the attention sink dynamics begin to compound, and the session drifts toward the wrong answer. In the ACM session, Archer calls:

```python
remove_context(
    message_ids=["msg-003", "msg-004", "msg-005"],
    summary=(
        "Race condition hypothesis INVALIDATED. "
        "Redis SET NX lock TTL timing windows too narrow for 1-in-40 failure rate. "
        "Lock acquisition path ruled out as cause."
    ),
    label="dead-end-A",
)
```

Now demonstrate the failure mode. ACM's failure is metacognitive error: the model misjudges what is important. In Archer's session, the specific failure looks like this: Archer, having read the task description and begun the race condition analysis, decides that the task description is now redundant — the constraints are embedded in its reasoning, it argues, so the original message can be removed to save space. It calls `remove_context(["msg-001"])` without pinning first. The Context View loses the load-dependency and staging-immunity constraints. The session continues. Later, when the clock-skew analysis is underway, Archer generates a hypothesis about a deterministic timing error in the JWT decoder — a hypothesis that the staging-immunity constraint would immediately rule out, because the JWT decoder behavior is identical in staging and production. But the constraint is no longer in the Context View. Archer pursues the deterministic hypothesis. It generates test scaffolding. The test fails, as it must, because the bug does not reproduce deterministically. Archer has lost two hours to a hypothesis that the original constraint would have eliminated in one step.

This is a real failure. ACM does not prevent it. What ACM does is make it visible, recoverable, and correctable in a way that no prior approach achieves. The tool call log shows, at turn 8, a `remove_context` call on `msg-001` without a prior `pin`. A human reviewer auditing the session log can see exactly when the constraint was removed, exactly what reasoning Archer provided in the summary field, and exactly why that reasoning was insufficient. `retrieve_context(message_ids=["msg-001"])` restores the constraint immediately. A subsequent agent pass, or a human intervention, can re-pin `msg-001` and continue the session from the point of restoration. The session is not lost. The error is not propagated silently through four hours of compounding contamination. It is a discrete, identifiable event in an auditable log, correctable at the turn where it occurred.

Compare this failure signature to the failures of the previous four approaches. Passive truncation produces a wrong answer with no tool call log, no removal event, no point of intervention — the contamination is continuous and distributed across the entire session history, and a forensic analyst examining the session cannot identify the turn where the failure began because there was no turn where the failure began; it was structural from the start. Compaction-on-threshold produces a wrong answer with a compaction event in the log, but the compaction was performed by the engine without semantic judgment, and the log records that it fired, not that it was wrong to fire. LCM produces a wrong answer with a rich provenance record in the Immutable Store, but the provenance record shows what was compacted, not why, and the model was not consulted about the decision. ACM produces a potentially wrong answer with a specific, labeled, timestamped tool call that shows exactly which messages were removed, what the model's stated reasoning was, and how to reverse the decision. The failure mode is the same category — the wrong content left the Active Context — but the failure signature is categorically different. In the first four approaches, the failure is silent. In ACM, the failure is auditable.

The architectural bet behind ACM is explicit: models are good enough at metacognitive judgment — at reasoning about what their own working context needs — to be trusted with write access to that context, and this capability will improve as models improve. LCM's architectural bet is the opposite: models are not yet reliable enough for this meta-task, so the engine should handle curation deterministically. Both bets are coherent. Both bets produce systems that work. The question is which failure mode an engineering team can live with. Silent, unauditable context rot that produces confident wrong answers and diagnoses as model unreliability — or visible, recoverable metacognitive errors that surface in tool call logs and invite targeted correction. ACM's bet is that the second failure mode is preferable, not because it never occurs, but because it is the only one of the five failure modes that a system can detect, recover from, and learn from in production.

### The Tetrahedron Read

**Structure** in ACM is the cleanest separation of the five approaches: an immutable Conversation Log that records everything without exception, and a Context View that is a computed, ephemeral projection of that log, rebuilt fresh on every turn from the accumulated `pin`, `remove_context`, and `retrieve_context` directives the model has issued. The structural consequence of this separation is that the Context View can be modified between turns without any operation on the log — no row is updated, no node is rewritten, no entry is discarded. The log is a ledger; the view is a query result. This structure gives ACM a property that none of the previous approaches possess: the Context View at any turn is fully auditable by replaying the directive sequence from the beginning of the session. An engineer who wants to understand why Archer's Context View at 2:30 PM contained what it contained can replay every `pin`, `remove_context`, and `retrieve_context` call in sequence and reconstruct the view at any prior turn exactly. The structure is not just recoverable; it is reproducible. That reproducibility is the basis of ACM's auditability guarantee, and auditability is the basis of the claim that ACM's failure mode is correctable in a way that the others are not.

**Logic** under ACM operates over a Context View that is, in the best case, exactly what the current state of the task requires: the pinned constraints that must never leave, the current hypothesis under investigation, the tombstone summaries of invalidated dead-ends, and the recent tool results that bear on the current analysis. In Archer's ACM session after the `remove_context` call at 11:46 AM, the Context View contains approximately one-tenth the token count of the passive session at the same point, and every token in it is either a pinned constraint, a tombstone summary, or current active reasoning. The attention computation runs over this reduced, curated context, and the attention sink dynamics that contaminated the passive session do not apply: there are no eleven instances of `lock_acquired_at` in the Context View, because Archer removed them. The logic that follows from this context is cleaner, more constrained, and more likely to converge on the correct hypothesis — not because the model is smarter, but because the substrate of its reasoning has been curated by an entity that understands what the reasoning requires. Logic quality in ACM is a function of the model's metacognitive accuracy: the better the model is at judging what the task needs, the cleaner the Context View, and the more reliable the reasoning. This is the dimension on which ACM will improve as models improve. The honest current picture is that metacognitive error rates in ACM-style deployments are not yet well-measured in published literature — which means an engineering team adopting ACM should instrument and monitor the error rate from first deployment rather than assuming it is low. The passive approaches have no equivalent measurement because their failures do not produce discrete events. That asymmetry — known error rate versus structurally unobservable failure — is itself the architectural argument, but it should not be mistaken for evidence that ACM's error rate is zero.

**Implementation** of ACM is lighter than LCM on infrastructure and heavier than passive truncation on model capability requirements. The Conversation Log requires append-only persistent storage — simpler than LCM's PostgreSQL-backed DAG, because the log is a flat sequence rather than a hierarchical graph, and provenance tracking is implicit in the sequential IDs rather than requiring explicit parent-child relationships. The Context View computation requires a directive replay mechanism: on each turn, the engine reads the accumulated directives and applies them to the full log in order to produce the current view. The four tool definitions must be exposed to the model through whatever tool-calling interface the inference API supports. The model must be prompted or fine-tuned to use these tools proactively and accurately — to pin constraints at session initialization, to remove dead-ends at the moment of invalidation rather than after contamination has already compounded, and to write tombstone summaries specific enough to be useful if the content is later retrieved. That last requirement is where implementation complexity lives: not in the infrastructure, but in the model behavior. A model that uses `remove_context` reactively — calling it only when the context is already crowded — will capture some of ACM's benefit while missing its full value. A model that uses `remove_context` proactively — at the moment a hypothesis is invalidated, before the scaffolding has had time to become an attention sink — captures the benefit in full. Implementing ACM well is, in part, a prompt engineering and fine-tuning problem, not only an infrastructure problem.

**Outcome** of ACM in Archer's session, when the model's metacognitive judgment is correct, is a five-hour debugging session that delivers the right patch. The task constraints survive the session unpinned and inviolable. Each dead-end is replaced by a specific tombstone at the moment of invalidation. The clock-skew analysis proceeds on a Context View composed entirely of currently-relevant content. The attention sink dynamics that contaminated every prior approach are absent because the tokens that would have created them were removed before they could accumulate. The final Context View, at 3:00 PM, is not a four-hour accumulation of every reasoning step Archer took — it is a curated working memory that reflects what the task currently requires: the original constraints, the tombstones of ruled-out hypotheses, and the live analysis of the clock-skew path. Archer presents the correct patch. The 401 errors stop. When the outcome is wrong — when Archer has made a metacognitive error and removed something it needed — the outcome is a recoverable session with an auditable failure point rather than a silent wrong answer with no diagnostic surface. Five approaches. Five answers to the same question about who controls the active context, by what mechanism, and with what failure signature. The matrix that follows maps each answer to its cost.

## The Decision Matrix

| Approach | Who curates | Selection criterion | Failure mode | Failure visibility | Infrastructure cost |
|---|---|---|---|---|---|
| Passive | Nobody | None (accumulate all) | Accumulation — stale KV cache entries persist indefinitely | Silent | Minimal — append loop only |
| Compaction-on-threshold | Engine | Oldest N messages | Semantic blindness — oldest ≠ least relevant; critical early constraints compressed before dead-end scaffolding | Silent | Low — threshold + summarization call |
| RLM | Model (once, at init) | Model-written code over fixed input | Problem mismatch — batch architecture applied to streaming conversational session | Silent | Medium — REPL environment, LLMProxy, metadata extraction |
| LCM | Engine | Oldest compactable DAG node | Semantic blindness by exclusion — model has no voice in curation; lossless ≠ semantically current | Silent | High — persistent DAG, PostgreSQL, provenance tracking, three-level escalation |
| ACM | Model (continuously) | Model's semantic judgment | Metacognitive error — model removes what it will need, or fails to pin what is critical | **Auditable** | Medium — append-only log, directive replay, four tool definitions |

![Figure 5 — Failure Mode Visibility Matrix](figures/figure5_matrix.png)
*Figure 5: ACM is the only approach whose failure mode a human can see, recover from, and learn from in production.*

The right row for a given workload is determined by three signals: session length, dead-end frequency, and reversibility requirement. For sessions shorter than ten to fifteen turns on a focused, single-hypothesis task — a code generation request, a one-step retrieval, a structured transformation — passive truncation is correct. The context never has time to rot, the implementation overhead of any active strategy is not justified, and the failure mode does not activate within the session's natural envelope. For sessions of moderate length where dead-ends are rare and the task is well-scoped, compaction-on-threshold provides a meaningful safety margin over passive without requiring model participation. It is the right default for workloads where the primary risk is token budget exhaustion rather than semantic contamination — long-document summarization, multi-step extraction pipelines, structured report generation where each turn advances a single thread rather than branching into competing hypotheses. RLM is the right choice when the problem is genuinely one of input size rather than session-generated content: processing a multi-million-token codebase to answer a specific question, navigating a corpus too large to fit in any context window, performing needle-in-haystack retrieval over a fixed document set. LCM is the right choice when recoverability and convergence guarantees are non-negotiable and the organization can absorb the infrastructure cost — regulated environments where session provenance must be auditable, or systems where the cost of losing early constraints is high enough to justify the DAG and its maintenance overhead.

ACM is the right choice for exactly the class of problem Archer was solving: long-running, exploratory, multi-hypothesis sessions where dead-end frequency is high, where the session generates its own content rather than processing a fixed input, and where the cost of a silent wrong answer exceeds the cost of an auditable recoverable one. The workload signals that indicate ACM are: sessions expected to run longer than thirty turns, tasks with open-ended hypothesis spaces where the agent cannot know in advance which branches will prove valid, domains where early constraints are highly specific and their loss is catastrophic (failure rates, environment dependencies, performance thresholds), and production systems where a human or supervisory agent will review session logs and must be able to reconstruct the agent's curation decisions. The last signal is the most important: ACM is the only approach whose failure mode is compatible with human oversight, because it is the only approach that produces a failure mode a human can see. Architecture that produces auditable failures is architecture that can be supervised, corrected, and improved. Architecture that produces silent failures can only be replaced.
